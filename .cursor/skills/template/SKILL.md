# Eddy Core Templates

This repo contains curated workflow templates available to all Eddy users from the "New Workflow" form. The Eddy API fetches from this repo at runtime to serve templates.

## Repo Structure

```
eddy-templates/
  manifest.json              # Auto-generated — do NOT edit by hand
  scripts/
    build-manifest.js        # Generates manifest.json from template files
  templates/
    simple-onboarding.json   # One file per template
```

## Template JSON Format

Each template file is a standard **`EddyWorkflowExportT`** — the same JSON format produced by Eddy's workflow export feature. Key top-level fields:

- `version` — export format version (currently `1`)
- `workflow` — workflow definition including `name`, `description`, `style`
- `pages` — stages in the workflow (with `title`, `node.position`, `is_starting`, `is_end`)
- `sections` — page sections
- `blocks` — input blocks, content blocks, etc.
- `blockOptions` — options for select/checklist blocks
- `pageTransitions` — edges between pages (`source_id`, `target_id`, optional `rule`)
- `workflowRoles` — role definitions
- `stageRoleAssignments` — which roles can access which stages
- `sheets` — associated sheet definitions
- `columns` — sheet column definitions

Templates are created by **exporting a real workflow** from the Eddy app — do not hand-author template JSON files.

## Adding a Template

1. In the Eddy app, build and publish the workflow you want to template
2. Export it using the export feature
3. Place the JSON file in `templates/` — the filename becomes the **slug** (e.g. `content-approval.json` → slug `content-approval`)
4. Ensure `workflow.name` and `workflow.description` are set in the JSON
5. Run: `node scripts/build-manifest.js`
6. Commit both the new template file and the updated `manifest.json`

## Editing a Template

To update a template's name, description, or other metadata, edit the relevant field in the template JSON file directly, then rebuild the manifest:

```bash
node scripts/build-manifest.js
```

For structural changes (adding pages, blocks, transitions), re-export the workflow from Eddy and replace the file.

## Build Script

`scripts/build-manifest.js` auto-generates `manifest.json` by scanning `templates/` for JSON files. For each template it:

1. Derives **slug** from the filename
2. Reads **name** from `workflow.name` and **description** from `workflow.description`
3. Sets **schemaVersion** to `1`
4. Extracts **preview** data for `WorkflowMiniGraph` rendering:
   - `pages`: `id`, `name`, `position`, `is_start_stage`, `is_end_stage`
   - `pageTransitions`: `source_id`, `target_id`
   - `orientation`: from `workflow.style.orientation` (defaults to `"vertical"`)
5. Sets the **file** path to `templates/<filename>.json`
6. Sorts templates alphabetically by name

Always run the build script after any template change. The manifest must stay in sync with the template files.

## Naming Conventions

- **Filenames**: lowercase, hyphen-separated (e.g. `employee-onboarding.json`)
- **Slugs**: derived automatically from the filename (no manual override)
- **Workflow names**: title case, human-readable (e.g. "Employee Onboarding")

## How the System Consumes Templates

The Eddy API (in the main `app/` codebase) fetches from this repo at runtime:

1. `GET /api/workflows/templates` — fetches `manifest.json`, returns template list with preview data (no full JSON)
2. `GET /api/workflows/templates/:slug` — fetches the full template JSON by slug
3. The client POSTs the full JSON to `POST /api/workflows/import` to hydrate it into the user's workspace

Both fetches are cached server-side with a TTL and stale-on-failure fallback. See `app/plans/core-templates-system.md` for the full architecture.

## Key Constraints

- **Do not edit `manifest.json` by hand** — it is always generated by the build script
- **Do not hand-author template JSON** — always export from Eddy to ensure format validity
- **Merge-to-main is live** — changes to `main` are served to users after the API cache TTL expires
- **Schema version**: all v1 templates use `schemaVersion: 1`; only increment when the payload contract changes
